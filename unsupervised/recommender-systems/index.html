Recommender Systems

Lecture notes · March 2026

💡 Why Recommender Systems Matter

  • Recommender systems power suggestions on online shopping (Amazon), video streaming (Netflix), food delivery apps, social media feeds, and more.
  • For many companies, a large fraction of total sales is driven directly by their recommender system.
  • The commercial and economic value of recommender systems is arguably vastly greater than the academic attention they receive.
  • The same core logic applies across domains: products, movies, restaurants, articles, ads.

🎬 Problem Setup — Movie Ratings Example

  • Running example: a movie streaming website where users rate movies 0–5 stars.
  • Users: Alice, Bob, Carol, Dave (users 1–4). Movies: Love at Last, Romance Forever, Cute Puppies of Love, Nonstop Car Chases, Sword vs. Karate (movies 1–5).
  • Not every user rates every movie — missing ratings are denoted with ?.

🎯 Goal & Approach

  • Goal: For every (user, movie) pair where r(i,j) = 0, predict what rating the user would give. Use those predictions to recommend movies the user is likely to rate highly (5 stars).
  • First approach (this week): Assume we have access to features about the movies — e.g. how much of a romance film it is, how much of an action film it is. Use these features to build the algorithm.
  • Later this week: Revisit the algorithm for the more realistic case where we do not have pre-built features — and see how it can still work.
Key insight: The same algorithm that works for movies works for any item — products, restaurants, articles, ads — as long as you have users, items, and some ratings or interaction signal.

🎞️ Using Movie Features to Predict Ratings

  • Suppose each movie i has a feature vector X(i) describing its content. For example:
    • x1 = romance score (0–1)
    • x2 = action score (0–1)
  • n = number of features (here n = 2).

Sword vs. Karate0.00.9
  • For each user j, predict the rating of movie i as:

    ŷ(i,j) = w(j) · X(i) + b(j)

    This is essentially linear regression — one separate model per user.
  • Example: Alice (user 1) with w(1) = [5, 0], b(1) = 0. Prediction for Cute Puppies of Love (X = [0.99, 0]):

    5 × 0.99 + 0 × 0 + 0 = 4.95 ✓ (plausible — Alice loves romance films)

💰 Cost Function — Single User j

  • m(j) = number of movies rated by user j.
  • Only sum over movies i where r(i,j) = 1 (user j has actually rated movie i).
  • Cost function (with regularisation):

    J(w(j), b(j)) = (1/2m(j)) Σ{i:r(i,j)=1} (w(j)·X(i) + b(j) − y(i,j))² + (λ/2m(j)) Σk (wk(j))²

  • Minimising this over w(j) and b(j) gives good parameters for predicting user j's ratings.
  • In practice, the 1/m(j) normalisation term is often dropped — it is just a constant and does not change the optimal w and b.

👥 Cost Function — All Users

  • To learn parameters for all nu users simultaneously, sum the per-user cost over j = 1 … nu:

    J(w(1),b(1),…,w(nu),b(nu)) = Σj=1nu [ (1/2m(j)) Σ{i:r(i,j)=1} (w(j)·X(i) + b(j) − y(i,j))² + regularisation ]

  • Minimise this combined cost using gradient descent or any other optimisation algorithm to obtain all parameters at once.
  • This is equivalent to training nu independent linear regression models, one per user, sharing the same movie feature vectors X(i).
Limitation: This approach requires pre-built movie features (e.g. romance score, action score). In practice, these are hard to obtain. The next topic removes this requirement.

🔎 Learning Movie Features from Data

  • What if we don't know the feature values x1, x2 (romance/action scores) in advance? We can learn them from the ratings data — but only if we already have parameters w and b for the users.
  • Intuition: If we know that Alice (w(1) = [5,0]) rated movie 1 as 5 stars, then w(1) · x(1) ≈ 5 implies x(1) ≈ [1, 0] (very romance, no action). Multiple users constrain what x must be.
  • This only works because multiple users have rated the same movies — with just one user there is not enough signal to recover the features from scratch.

📐 Cost Function — Learning Features x(i)

  • Given user parameters w(j), b(j) for all j, the cost for learning the feature vector of a single movie i is:

    J(x(i)) = (1/2) Σ{j:r(i,j)=1} (w(j)·x(i) + b(j) − y(i,j))² + (λ/2) Σk (xk(i))²

  • Sum only over users j who have rated movie i (r(i,j) = 1).
  • To learn features for all nm movies, sum this cost over i = 1 … nm.

🤝 Collaborative Filtering — Combined Cost Function

  • The cost function for learning w and b (previous section) and the cost function for learning x share the same squared-error term. Combining them gives:

    J(w, b, x) = (1/2) Σ(i,j): r(i,j)=1 (w(j)·x(i) + b(j) − y(i,j))²
                  + (λ/2) Σj,k (wk(j))² + (λ/2) Σi,k (xk(i))²

  • The sum runs over all (i,j) pairs where a rating exists — equivalent whether you sum over users first or movies first.
  • Minimise this jointly over all three sets of parameters: w, b, and x.
  • Note: x is now a parameter to be learned, not a fixed input — unlike standard linear regression.
Key insight: Summing over users first or movies first covers exactly the same set of rated (i,j) pairs — the two formulations are equivalent.

📉 Optimisation via Gradient Descent

  • Apply gradient descent (or another optimiser) with respect to all parameters simultaneously:

    wk(j) := wk(j) − α · ∂J/∂wk(j)
    b(j)    := b(j)    − α · ∂J/∂b(j)
    xk(i)  := xk(i)  − α · ∂J/∂xk(i)

  • Updating all three simultaneously converges to a good set of user parameters and movie features.

🌐 Why "Collaborative Filtering"?

  • Multiple users have rated the same movies collaboratively — together they provide enough signal to infer what the movies are like (the features x).
  • Those learned features then allow the algorithm to predict ratings for users who have not yet seen those movies.
  • Users effectively collaborate — even without knowing each other — to help the system make better recommendations for everyone.

👍 Binary Labels in Collaborative Filtering

  • Instead of 1–5 star ratings, many real-world systems only have binary feedback: did the user engage with the item or not?
  • Label meanings:
    • 1 — user engaged after being shown the item (watched to the end, clicked, liked, favourited, purchased, spent ≥ 30 seconds).
    • 0 — user was shown the item but did not engage.
    • ? — user has not yet been shown the item.

🔀 Adapting the Model for Binary Labels

  • Star-rating model (regression):

    ŷ(i,j) = w(j) · x(i) + b(j)

  • Binary-label model (logistic): predict the probability that y(i,j) = 1:

    P(y(i,j) = 1) = g(w(j) · x(i) + b(j))

    where g(z) = 1 / (1 + e−z) is the logistic (sigmoid) function.
  • This mirrors the transition from linear regression to logistic regression in course 1 — same idea applied to the collaborative filtering setting.

💰 Cost Function — Binary Labels

  • The squared-error cost is replaced by the binary cross-entropy loss used in logistic regression:

    L(f, y) = −y · log(f) − (1 − y) · log(1 − f)

    where f = g(w(j) · x(i) + b(j)).
  • The full cost function over all rated pairs becomes:

    J(w, b, x) = Σ(i,j): r(i,j)=1 L(f(x(i), w(j), b(j)), y(i,j)) + regularisation

  • The regularisation terms (λ/2 · Σw² and λ/2 · Σx²) remain the same as before.
  • Minimise this over w, b, and x using gradient descent — the structure of the algorithm is otherwise identical to the star-rating version.
Impact: This generalisation dramatically expands the range of applications: any system that logs clicks, purchases, likes, or watch-time can now be addressed with collaborative filtering.

⚖️ Mean Normalisation

  • For recommender systems with numerical ratings (0–5 stars), mean normalisation makes the algorithm run faster and produce much better predictions for new or sparse users.
  • The problem without it: A new user Eve who has rated no movies contributes nothing to the squared-error term. Regularisation pushes w(5) → [0, 0] and b(5) → 0, so the algorithm predicts 0 stars for every movie — unhelpful.

🧮 Mean Normalisation — Procedure

  1. Collect all ratings (including ?s) into a matrix Y of shape nm × nu.
  2. For each movie i, compute μi = average rating across all users who have rated it:

    μ = [2.5, 2.5, 2.0, 2.25, 1.25] (for the five-movie example)

  3. Subtract μi from every rating of movie i:

    Y'(i,j) = Y(i,j) − μi (for all j where r(i,j) = 1)

    e.g. a 5-star rating for movie 1 becomes 5 − 2.5 = 2.5; a 0-star rating becomes 0 − 2.25 = −2.25.
  4. Train the collaborative filtering algorithm on the normalised ratings Y'.
  5. When predicting, add μi back:

    predicted rating = w(j) · x(i) + b(j) + μi

👤 Effect on New / Sparse Users

  • With mean normalisation, a new user Eve still learns w(5) = [0,0] and b(5) = 0 (no ratings to learn from), but her predicted rating for movie i becomes:

    0 · x(i) + 0 + μi = μi

    i.e. the algorithm predicts the average rating other users gave that movie — much more sensible than 0 stars for everything.
  • As Eve rates movies over time, her w and b will be updated away from zero and her predictions will personalise.
Key benefit: Mean normalisation acts as a warm start — new users get the crowd average instead of zero, making early recommendations immediately reasonable.

📊 Rows vs. Columns — What to Normalise

  • Normalise rows (movies): Handles new users with few or no ratings. Recommended approach.
  • Normalise columns (users): Would help if a brand-new movie has no ratings yet. Less important in practice — a movie with no ratings probably shouldn't be shown to many users yet anyway.
  • For this week's practice lab, normalising rows only is sufficient.

⚙️ Implementing Collaborative Filtering in TensorFlow

  • TensorFlow is not just for neural networks — it is also a great tool for implementing other learning algorithms like collaborative filtering.
  • The key benefit: automatic differentiation (Auto Diff). You only need to implement the cost function J; TensorFlow computes all derivatives for you automatically.
  • This removes the need to manually derive and code partial derivatives with respect to w, b, and x.
  • Once you have automatic gradients, you are also free to use more powerful optimisers beyond plain gradient descent — such as the Adam optimiser.

📼 Auto Diff — Gradient Tape

  • TensorFlow's gradient tape records the sequence of operations used to compute J, then automatically differentiates through them.
  • Simple example — minimising J = (wx − 1)²:

    w = tf.Variable(3.0)                  # declare w as a trainable variable
    with tf.GradientTape() as tape:
        fwx = w * x
        J = (fwx - y) ** 2
    dJdw = tape.gradient(J, w)          # derivative computed automatically
    w.assign_add(-alpha * dJdw)         # gradient descent update

  • TensorFlow knows to differentiate with respect to w because it was declared as a tf.Variable.
  • Auto Diff is the technically correct term; Auto Grad refers to a specific software package but is often used loosely for the same concept. PyTorch also supports this feature.

🧩 Full Collaborative Filtering — TensorFlow Recipe

  • Declare x, w, b as tf.Variables (parameters to be learned).
  • Use the Adam optimiser instead of plain gradient descent:

    optimizer = keras.optimizers.Adam(learning_rate=1e-1)

  • For each iteration:
    1. Use tf.GradientTape to compute the cost J(x, w, b) — the normalised collaborative filtering cost function.
    2. Call tape.gradient(J, [x, w, b]) to get all three gradients at once.
    3. Call optimizer.apply_gradients(zip(grads, [x, w, b])) to update all parameters.
  • The cost function J takes as inputs: x, w, b, the mean-normalised ratings Y_norm, the indicator matrix r, nu, nm, and the regularisation parameter λ.
Why not use model.fit? The collaborative filtering cost function does not fit the standard dense-layer structure, so the usual model.compile / model.fit recipe cannot be used. Instead, we implement J manually and let TensorFlow handle the rest via Auto Diff.

📉 Gradient Descent Update Rules (Reference)

  • If using plain gradient descent instead of Adam, the updates at each iteration are:

    w(j) := w(j) − α · ∂J/∂w(j)
    b(j) := b(j) − α · ∂J/∂b(j)
    x(i) := x(i) − α · ∂J/∂x(i)

  • All three sets of parameters are updated simultaneously each iteration.
  • With TensorFlow's Auto Diff you never need to derive these partial derivatives by hand.
Coming next: How to find related items — given one movie, how do you find other movies similar to it using the learned feature vectors?

🔗 Finding Related Items

  • After training, every item i has a learned feature vector x(i). These features collectively describe what the item is like — even if individual features are hard to interpret on their own.
  • To find items similar to item i, find item k whose feature vector x(k) is closest to x(i) using the squared distance:

    ‖x(k) − x(i)‖² = Σl=1n (xl(k) − xl(i))²

  • Find the 5–10 items with the smallest squared distance to x(i) → these are the most related items.
  • This is how websites show "customers also viewed" or "movies similar to this one."
Note: The learned features x(i) are often hard to interpret individually, but collectively they capture meaningful properties of each item that make distance a useful similarity measure.

⚠️ Limitations of Collaborative Filtering

  • Cold start problem:
    • New item: A newly published movie with very few ratings is hard to rank accurately — there is not enough signal in the ratings yet.
    • New user: A user who has rated only a few items will get less personalised recommendations. Mean normalisation helps, but the problem remains.
  • Side information is not naturally used: Collaborative filtering has no built-in way to incorporate extra knowledge you may already have, such as:
    • About movies: genre, cast, studio, budget, release year.
    • About users: age, gender, location (inferrable from IP), expressed genre preferences, device type (mobile vs desktop), even web browser — all of which can be surprisingly predictive of preferences.
Coming next: Content-based filtering — a state-of-the-art technique that directly addresses both limitations by incorporating rich side information about users and items.

⚖️ Collaborative vs. Content-Based Filtering

Collaborative Filtering Content-Based Filtering
Core idea Recommend items based on ratings from users who rated things similarly to you Recommend items by matching features of users with features of items
Inputs Only ratings r(i,j) and y(i,j) Ratings plus rich feature vectors for each user and each item
Cold start Struggles with new users / new items Can handle new users / items if features are available
Side information Not used Directly incorporated into feature vectors

📋 User & Item Feature Vectors

  • User features xu(j) for user j may include:
    • Age, gender (one-hot), country (one-hot, ~200 values).
    • Which of the top-1,000 movies the user has already watched (1,000 binary features).
    • Average rating the user has given per genre (e.g. average romance rating, average action rating) — derived from their own ratings.
  • Item features xm(i) for movie i may include:
    • Year of release, genre(s).
    • Critic review sentiment features.
    • Average rating of the movie overall, or per country / demographic.
  • User and item feature vectors can be different sizes — e.g. xu may have 1,500 numbers while xm has only 50.
  • Features derived from ratings (e.g. average genre rating) are perfectly valid even though they depend on user behaviour.

🤝 The Content-Based Filtering Model

  • The goal is to compute two compact vectors from the raw features:
    • vu(j) — a vector summarising user j's preferences, computed from xu(j).
    • vm(i) — a vector summarising movie i's properties, computed from xm(i).
  • Predicted rating of user j for movie i:

    ŷ(i,j) = vu(j) · vm(i)

    (The b(j) bias term from collaborative filtering is dropped — it has negligible effect here.)
  • Critical constraint: vu and vm must have the same dimension (e.g. both 32 numbers) so that the dot product is defined — even though the raw input features xu and xm can be different sizes.
Intuition: If vu = [4.9, 0.1, …] encodes "loves romance, dislikes action" and vm = [4.5, 0.2, …] encodes "strongly romantic, barely action", their dot product will be large — correctly predicting a high rating.

🧠 Deep Learning for Content-Based Filtering

  • Two separate neural networks are used to compute the compact vectors:
    • User network: takes xu (age, gender, country, …) as input → outputs vu (e.g. 32 numbers).
    • Movie network: takes xm (release year, cast, genre, …) as input → outputs vm (e.g. 32 numbers).
  • The output layer of each network has 32 units (not 1 unit as in a typical regression network). Both output layers must have the same dimension so that a dot product can be taken.
  • The two networks may have different numbers of hidden layers and units per layer — only the output size must match.
  • Predicted rating:

    ŷ(i,j) = vu(j) · vm(i)

    For binary labels, wrap in the sigmoid function: g(vu(j) · vm(i)).
Architecture note: The two networks can be drawn as a single combined diagram — user features feed into the top half, movie features into the bottom half, and the dot product of both outputs gives the prediction.

🏋️ Training the Combined Network

  • Both networks are trained jointly using a single cost function — there is no separate training procedure for each:

    J = Σ(i,j): r(i,j)=1 (vu(j) · vm(i) − y(i,j))² + regularisation

  • Gradient descent (or Adam) tunes the parameters of both networks simultaneously so that the dot products match the observed ratings as closely as possible.
  • Standard neural network regularisation can be added to keep parameter values small and prevent overfitting.
  • This is an example of putting two neural networks together to work in concert — one of the strengths of the neural network paradigm over other ML approaches.

🔗 Finding Similar Items with Content-Based Filtering

  • After training, each movie i has a learned vector vm(i) of length 32 that encodes its properties.
  • To find movies similar to movie i, look for movie k with the smallest squared distance:

    ‖vm(k) − vm(i)‖²

  • Pre-computation: Because these distances depend only on the movie vectors (not on who is browsing), you can run an overnight compute job to pre-compute the top-10 or top-20 similar movies for every movie in the catalog. When a user visits a movie page, the results are already ready — no real-time computation needed.
  • Pre-computation becomes especially important at scale when the catalog contains millions of items.

⚡ Scaling Up — Retrieval & Ranking

  • Large-scale catalogs make it infeasible to run full neural network inference on every item for every user visit:
    • Movie streaming sites: thousands of movies.
    • Ad systems: millions of ads.
    • Music streaming: tens of millions of songs.
    • Online shopping: millions to tens of millions of products.
  • Solution: a two-stage pipeline — Retrieval then Ranking.

🔍 Stage 1 — Retrieval

  • Quickly generate a large list of plausible candidates — broad coverage matters more than precision here. It is okay to include items the user might not like.
  • Example retrieval strategies (combined):
    • For each of the last 10 movies the user watched, pull the 10 most similar movies (using pre-computed vm distances) → up to 100 candidates.
    • For the user's top-3 most-watched genres, add the top-10 movies per genre → up to 30 more.
    • Add the top-20 movies in the user's country → 20 more.
  • Combine all retrieved items into one list; remove duplicates and items the user has already watched or purchased.
  • Finding similar movies is pre-computed (look-up table) — retrieval is therefore very fast.

🏆 Stage 2 — Ranking

  • Take the retrieved list (e.g. ~100–500 items) and rank them using the full neural network model.
  • Efficient inference trick: Pre-compute vm for all movies offline. At serving time:
    1. Run the user network once to compute vu for the current user.
    2. Take the dot product of vu with the pre-computed vm vectors of only the retrieved candidates.
    Full network inference is only needed once per user visit, not once per item.
  • Rank retrieved items by predicted rating (or probability of engagement) and display the top results.
Key benefit: Retrieval prunes millions of items down to hundreds cheaply. Ranking then applies the accurate model to only those hundreds — giving both speed and accuracy.

🎛️ Tuning Retrieval Size

  • Retrieving more items → better ranking quality, but slower pipeline.
  • Use offline experiments to find the sweet spot: measure how much the predicted ratings of top results improve as you increase retrieval size (e.g. 100 vs. 500 vs. 1,000 items).
  • If retrieving 500 instead of 100 items substantially increases the estimated probability that recommended items are ones the user will actually like, the extra compute cost may be justified.

⚖️ Ethical Considerations in Recommender Systems

  • Recommender systems are highly profitable — but some use cases have left users and society worse off.
  • Common optimisation targets and their implications:
What is optimisedAssessment
Movies most likely to be rated 5 starsGenerally fine — directly serves user preferences
Products most likely to be purchasedGenerally fine — aligns user and business interests
Ads most likely to be clicked (weighted by bid)Profit-maximising; can amplify exploitative businesses
Products that generate the largest profit marginMay not show users the most relevant items; lack of transparency
Content that maximises watch time / engagementCan amplify conspiracy theories, hate, and toxic content

📢 The Advertising Amplification Problem

  • Ad-based recommender systems create a feedback loop: the more profitable a business, the higher it can bid for ads, the more traffic it receives, and the more profitable it becomes.
  • Virtuous cycle (good business): A travel company that genuinely serves customers well → more profit → higher ad bids → more traffic → more customers served well.
  • Vicious cycle (harmful business): A payday loan company that exploits customers → more profit from exploitation → higher ad bids → more traffic → more customers exploited.
  • The same feedback mechanism that rewards good businesses can also reward the most exploitative ones — this is a fundamental tension in ad-funded recommender systems.
Possible amelioration: Refuse to serve ads from exploitative businesses — but defining what counts as "exploitative" is genuinely difficult and requires open debate and diverse perspectives.

🔥 Engagement Maximisation & Harmful Content

  • Optimising for watch time or time-on-site has been widely reported to cause large platforms to amplify:
    • Conspiracy theories
    • Hate speech and toxic content
    because such content is highly engaging even when harmful to individuals and society.
  • Partial amelioration: Filter out hate speech, fraud, scams, and certain violent content before it enters the recommendation pool.
  • Defining exactly what to filter is surprisingly difficult — companies, individuals, and governments all need to continue working through these questions.

🔎 Transparency & Responsible Building

  • Many users assume a website recommends things it thinks they will enjoy — most do not realise recommendations are often optimised for profit rather than user satisfaction.
  • Recommendation: Be transparent with users about the criteria used to decide what to show them. Transparency builds trust and encourages systems to be designed for user benefit.
  • When building any recommender system (or any ML system), ask:
    • What benefits does this create?
    • What potential harms could it cause?
    • Have diverse perspectives been invited to weigh in?
Guiding principle: Only build things you genuinely believe will make people better off. The power and profitability of recommender systems makes thoughtful, ethical design all the more important.

💻 Implementing Content-Based Filtering in TensorFlow

  • Both the user network and the item (movie) network are built as sequential models with dense hidden layers (relu activation) and a 32-unit output layer.
  • User network sketch:

    user_NN = tf.keras.models.Sequential([
      Dense(256, activation='relu'),
      Dense(128, activation='relu'),
      Dense(32) # outputs v_u
    ])

  • Item network has the same structure, outputting vm (32 numbers).
  • After computing vu and vm, each vector is L2-normalised (scaled to length 1) — this makes the algorithm work noticeably better:

    vu = tf.linalg.l2_normalize(user_NN(user_input), axis=1)
    vm = tf.linalg.l2_normalize(item_NN(item_input), axis=1)

  • The dot product is computed using a dedicated Keras layer:

    output = tf.keras.layers.Dot(axes=1)([vu, vm])

  • The full model is assembled by declaring inputs (user features + item features) and the output (dot product):

    model = tf.keras.Model([user_input, item_input], output)

  • Compile and train with mean squared error cost function (for star ratings) or binary cross-entropy (for binary labels).
Key detail: L2-normalising vu and vm before taking the dot product constrains both vectors to the unit sphere, which stabilises training and improves recommendation quality.

📉 Optional · Principal Components Analysis (PCA) — Overview

  • PCA (Principal Components Analysis) is an unsupervised learning algorithm used primarily for data visualisation.
  • Core problem it solves: When a dataset has many features (10, 50, 1,000+), it is impossible to directly plot. PCA compresses the data down to 2 or 3 features so it can be visualised on a screen or paper.
  • PCA does not simply discard features — it finds new axes that capture the most variation in the data.

🚗 PCA — Intuition with Examples

  • Example 1 — Width barely varies: Car length (x1) varies a lot; car width (x2) barely varies (road constraints keep cars ~1.8m wide). PCA essentially keeps x1 and drops x2.
  • Example 2 — Correlated features: Car length (x1) and car height (x2) both vary significantly and are correlated — bigger cars tend to be both longer and taller. Dropping either alone would lose information.
  • PCA solution for Example 2: Find a new axis z that lies within the x1–x2 plane and points in the direction of most variation. The coordinate of each car on z captures "overall size" — one number instead of two, with most information preserved.
  • 3-D to 2-D: If three-dimensional data essentially lies on a thin flat pancake inside 3-D space, PCA finds the two axes z1 and z2 of that pancake, reducing 3 features to 2 without much loss.
Key idea: PCA replaces the original axes (x1, x2, …) with new axes (z1, z2, …) that capture the directions of greatest variance, then keeps only the top 2 or 3.

🌍 PCA — Country Development Example

  • Suppose each country has 50 features: GDP, per-capita GDP, Human Development Index, life expectancy, etc.
  • PCA reduces these 50 features to 2 features z1 and z2, which can then be plotted on a 2-D scatter chart.
  • The learned axes often have intuitive interpretations:
    • z1 ≈ total country size / total GDP (larger countries score higher).
    • z2 ≈ per-person economic activity / prosperity.
  • Examples on the plot:
    • USA — large country, high per-person activity → upper right.
    • Singapore — small country, high per-person activity → lower right.
    • Large country with low per-person activity → lower left but far right on z1.

🔭 Why Use PCA?

  • Visualisation: The primary use — plot high-dimensional data to understand its structure at a glance.
  • Anomaly detection: Visualising the data can reveal unexpected patterns, outliers, or "something funny going on."
  • Dimensionality reduction: Reducing features from "50-dimensional" to "2-dimensional" (these terms just mean 50 features vs. 2 features).
  • PCA is routinely one of the first things a data scientist runs on a new dataset to get a feel for the data before modelling.
Coming next: How the PCA algorithm actually works — finding the new axes and projecting data onto them.

⚙️ How PCA Works

  • Preprocessing (required before PCA):
    • Subtract the mean from each feature so all features have zero mean.
    • If features are on very different scales (e.g. house size in sq ft vs. number of bedrooms), apply feature scaling first.
  • PCA then finds a new axis z such that when the data is projected onto z, the projected points retain the maximum possible variance (spread).
  • Projection means dropping a perpendicular line segment from each data point to the axis z — the foot of that perpendicular is the point's coordinate on z.

📏 Choosing the Principal Component

  • Different axis orientations produce different amounts of spread in the projected points:
    • A bad axis squishes all projections close together → low variance → little information retained.
    • A good axis keeps projections spread apart → high variance → most information retained.
  • The principal component (first principal component) is the axis orientation that maximises variance in the projections. This is the axis PCA chooses when reducing to 1 dimension.
  • Computing the projection: Given a data point with coordinates (x1, x2) and a unit vector pointing along z (e.g. [0.71, 0.71]):

    z = x · unit_vector = x1×0.71 + x2×0.71

    Example: point (2, 3) → z = 2×0.71 + 3×0.71 = 3.55

📐 Multiple Principal Components

  • If you need more than one new axis (e.g. reducing to 2-D), each additional axis must be perpendicular (90°) to all previously chosen axes.
  • This ensures the axes capture independent directions of variation.
  • Example: 50 features → find 3 principal components → z1, z2, z3 are all mutually perpendicular.

⚖️ PCA vs. Linear Regression

Linear RegressionPCA
Type Supervised learning Unsupervised learning
Data Labelled (x and y) Unlabelled (x only)
Goal Predict y from x Find axes that preserve variance; reduce dimensions
Distances minimised Vertical distances to the fitted line (along the y-axis) Perpendicular distances to the chosen axis (treats all features equally)
Feature treatment y is special; x features are inputs All features treated equally
Rule of thumb: Use linear regression to predict y. Use PCA to reduce the number of features for visualisation or further processing.

🔁 Reconstruction

  • Given a compressed coordinate z, PCA can approximately recover the original features by multiplying z by the unit vector of the principal component:

    x̂ = z × unit_vector = 3.55 × [0.71, 0.71] ≈ [2.52, 2.52]

  • The original point was (2, 3); the reconstruction gives (2.52, 2.52) — a reasonable approximation from just one number.
  • Reconstruction is not exact because information is lost during compression, but it gives you back an approximation of the original space.
Coming next: Implementing PCA in Python using scikit-learn.

🐍 PCA in Code — scikit-learn

  • Steps to implement PCA:
    1. (Optional) Feature scale if features have very different ranges (e.g. GDP in trillions vs. small indices).
    2. Call pca.fit(X) — scikit-learn automatically performs mean normalisation internally, so you do not need to do it separately.
    3. Inspect pca.explained_variance_ratio_ to see how much variance each principal component captures.
    4. Call pca.transform(X) to project all training examples onto the new axes.
  • Reducing to 1 component:

    from sklearn.decomposition import PCA
    pca = PCA(n_components=1)
    pca.fit(X)
    print(pca.explained_variance_ratio_) # e.g. [0.992]
    Z = pca.transform(X) # each example becomes 1 number

    explained_variance_ratio_ = [0.992] means 1 axis captures 99.2 % of the variance in the original data.
  • Reducing to 2 components: Change n_components=2. The ratios will sum to 1 (e.g. [0.992, 0.008]) — z1 still captures 99.2 %, z2 captures the remaining 0.8 %.
  • When reducing n-D data to n-D (no actual compression), reconstruction is exact and you recover the original data perfectly.
Tip: Always check explained_variance_ratio_ before using the compressed data — if the top 2–3 components explain ≥ 95 % of variance, the compression is lossless enough for visualisation.

✅ PCA — When to Use It (and When Not To)

Use caseRecommended?Notes
Data visualisation ✅ Most common & recommended Reduce to 2 or 3 features to plot and understand your dataset
Data compression ⚠️ Much less common today Was useful when storage and bandwidth were expensive; modern infrastructure largely makes this unnecessary
Speeding up supervised learning ❌ Rarely recommended now Used to help older algorithms (e.g. SVMs) with 1,000+ features; modern deep learning handles high-dimensional data directly and PCA's own cost offsets any gain
Bottom line: In modern ML, the primary reason to reach for PCA is visualisation — reducing a high-dimensional dataset to 2 or 3 dimensions to gain intuition before or during modelling.
